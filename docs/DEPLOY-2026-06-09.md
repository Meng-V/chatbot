# Smart Chatbot — Release Deployment Guide (2026-06-09)

**For:** Rachel (ops)
**From:** Meng
**Target:** production server (`ulblwebp20`), nginx + Puppet, Node/Python/Docker available.

This deploys the current `main` to production. It assumes you have SSH +
sudo on the prod host and know where the app and the nginx static root
live; it does **not** assume any project context beyond that.

---

## 0. What this release changes (why it matters)

This is a substantial release. The two things that matter operationally:

1. **The bot now actually answers "how-to / policy / service" questions.**
   Prose retrieval (`search_kb`) was previously unwired in production, so
   the bot refused (or, worse, fabricated) for a large class of questions
   (printing, Adobe, special collections, room booking, borrowing). That's
   fixed.
2. **Citations now render in the UI** and every URL the bot emits is
   verified to be a real, retrieved source (no more made-up / 404 links).
   **This half lives in the frontend**, so a backend-only deploy will NOT
   show citations — the frontend MUST be rebuilt (Step 3).

Expected behaviour change: the bot **refuses more often on thin-evidence
questions** (by design — it now refuses rather than inventing an answer).
That is correct, not a regression.

## 2. Backend (Python)

Pull `main` and restart the backend. Use whichever run mechanism this
host actually uses — systemd unit or Docker container:

```bash
cd /opt/chatbot/current        # adjust to the real app path
git fetch origin && git checkout main && git pull origin main

# Python deps may have changed — reinstall inside the app's venv
# (or rebuild the image if the backend is containerized):
ai-core/venv/bin/pip install -r ai-core/requirements.txt
ai-core/venv/bin/prisma generate    # regenerate the Prisma client

# Restart — pick the one that matches this host:
sudo systemctl restart smartchatbot-backend          # if systemd
# docker compose up -d --build smartchatbot-backend  # if Docker
```

Backend listens on **port 8081** in production and serves Socket.IO at
`/smartchatbot/socket.io`.

**If `pip install` errors with `ModuleNotFoundError: starlette.requests`**
(version drift), run:
`ai-core/venv/bin/pip install --force-reinstall --no-cache-dir fastapi starlette`.

Confirm it came up:
```bash
curl -s http://localhost:8081/health/live      # -> {"status":"alive"}
```

## 3. Frontend (Node) — **MANDATORY, do not skip**

The citation display + all UI fixes are a compiled React bundle. Build it
and publish the `dist/` to wherever nginx serves the widget from.

```bash
cd /opt/chatbot/current/client
npm ci
npm run build          # produces client/dist/ (~900 KB); build verified clean
```

The bundle is built with base path `/smartchatbot/`, so its assets resolve
under that path. Publish `client/dist/` to the nginx document root that
serves `/smartchatbot/` (replace the previous build there). Example if
nginx serves a static dir:

```bash
sudo rsync -a --delete client/dist/ /var/www/smartchatbot/   # adjust target
```

> If the frontend is built/served via Docker or a Puppet-managed static
> path, do the `npm run build` inside that pipeline instead and ship the
> resulting `dist/` the same way you do today. The only hard requirement:
> the new `dist/` must replace the old one at the `/smartchatbot/` root.

## 4. Embedding cache (one-time, then automatic)

`ai-core/data/eval/classifier_embeddings.json` is a **332 MB** file
(gitignored — too big for GitHub) that the intent classifier loads at
startup. Without it the first request after a restart re-embeds ~5,500
exemplars live (~30–60 s + a small OpenAI cost) and can look like a hang.

Keep it in a persistent location and symlink it on every deploy:

```bash
# one-time: stash it outside the release dir
sudo mkdir -p /opt/chatbot/shared
# (move the existing 332MB file there if it isn't already)

# every deploy (add to the deploy script / Puppet exec):
mkdir -p /opt/chatbot/current/ai-core/data/eval
ln -sf /opt/chatbot/shared/classifier_embeddings.json \
       /opt/chatbot/current/ai-core/data/eval/classifier_embeddings.json
```

> Regenerate it only when Meng says exemplars changed:
> `cd ai-core && venv/bin/python scripts/eval_classifier_v38.py`.

## 5. nginx + env (Puppet-managed pieces)

Two small config items, both best applied through Puppet:

**(a) Forward health checks to the backend.** The frontend's liveness
probe and any synthetic monitor hit `/health/live`; nginx must proxy
`/health/*` to the backend on `:8081`:

```nginx
location /health/ {
    proxy_pass http://127.0.0.1:8081;
    proxy_set_header Host $host;
}
```
(`/health/live` is cheap; `/health` and `/health/ready` do live dependency
checks — fine for an ops dashboard, not for a per-request probe.)

**(b) Admin token (optional, enables the librarian review page).** Set
`ADMIN_API_TOKEN` in the backend's environment. If unset, the `/admin/*`
review surface stays unmounted (fail-closed — conversation logs stay
private), which is a safe default; set it only when you want that page.

Apply via Puppet, then `sudo systemctl reload nginx` (or let Puppet do it).

## 6. Post-deploy smoke test

Open the chat widget and ask two questions:

1. **`what time do you close`** → should return King Library's live hours
   today, with a small clickable `[1]` citation chip.
2. **`where can I get Adobe`** → should return the Software Checkout page
   (`lib.miamioh.edu/software/`) as a clickable source — **never a 404**.

If both answer with working citation chips, the deploy is good. (First
message after a restart may take ~20–40 s — cold start — then it's fast.)

Server-side check:
```bash
curl -s http://localhost:8081/health/ready    # 200 if Weaviate+Postgres+OpenAI+LibCal reachable
curl -s http://localhost:8081/smoketest/v2     # runs a canned question end-to-end
```

## 7. Rollback

If anything looks wrong, revert to the commit just before this release
(`e24fa81` — which still includes the critical 400-error fix) and restart:

```bash
cd /opt/chatbot/current
git reset --hard e24fa81
sudo systemctl restart smartchatbot-backend     # or restart the container
# rebuild + republish the frontend from that commit too if the UI looks broken:
cd client && npm ci && npm run build && sudo rsync -a --delete dist/ /var/www/smartchatbot/
```
