# Deploy & Log Guide — 2026-06-16 release

Smart Chatbot v2, prod server **ulblwebp20**. Backend runs under systemd
unit **`smartchatbot-backend`** on **port 8081**, serving Socket.IO at
`/smartchatbot/socket.io`. Logs go to **journald**.

> Two kinds of change ship differently:
> - **Data** (Postgres rows — e.g. the Wertz address) is already live the
>   moment the seed/SQL runs; no restart needed.
> - **Code** (Python/React) needs a **redeploy + restart** to take effect.
> This release is mostly code, so it requires a full deploy.

---

## What's in this release

| Area | Fix |
|---|---|
| ManualCorrection (D3) | The librarian "fix an answer without a deploy" layer was silently failing on every prod turn (Prisma client bound to the wrong event loop + a poisoned cache). Now opens a fresh client per load with a 60s TTL cache; admin writes bust the cache so a correction is live on the next turn. |
| Springshare observability | New `[Springshare:LibCal]` / `[Springshare:LibGuides]` console logging for every API call and OAuth token fetch, plus a **pre-flight health check** that prints a banner at startup (before serving traffic). Secrets are redacted. |
| Monitoring gaps | Per-tool Prometheus metrics are now actually recorded (`search_kb`, `get_hours`, `book_room`, …); `/health/ready` now probes **LibGuides** too; OpenAI calls (incl. embeddings) now log a `[OpenAI]` line. |
| Citation links | The inline `[n]` popover was unclickable on the embedded widget. Replaced with a small **"Sources" footer** under each answer — plain `<a target="_blank">` links that just navigate (right-click / copy work too). Inline `[n]` is now a static marker. |
| Cold-start freeze (the "I encountered an error" drops) | On the first message after each restart, the v2 deps were built lazily **on the event loop** (~5.5k exemplars + a 347MB cache + live embeds, ~16–30s). That blocked Uvicorn's Socket.IO heartbeat, so the browser dropped the connection and showed the generic error (no traceback — the backend never crashed). Now the deps are **warmed at startup** in a worker thread, before traffic — the freeze is off the request path entirely. |
| MakerSpace "3D printer?" | The MakerSpace is its own data row; the agent kept querying the King building row (no 3D printer in its equipment) and refused. Now deterministically looks up the MakerSpace row, so equipment questions answer "Yes …" with the MakerSpace source. |
| Booking / data | Real LibCal room booking behind a confirm gate; Wertz address corrected (data already applied). |

## Deploy steps (in order)

### 0. Push the code
```bash
git push origin main
```
The deploy script clones `origin/main`, so anything not pushed won't ship.

### 1. ⚠️ Verify the classifier embedding-cache symlink is in place — strongly recommended
The kNN intent classifier needs a ~347 MB embedding cache
(`ai-core/data/eval/classifier_embeddings.json`). It is **gitignored**, and
`deploy.sh` does a **fresh clone per build**, so the cache is NOT in the
clone. It must be symlinked in from the persistent location:

```bash
# persistent copy lives outside the build dirs:
ls -la /opt/chatbot/shared/classifier_embeddings.json   # should exist & be ~347MB

# symlink it into the new build (see docs/DEPLOY-2026-06-09.md §4):
ln -sf /opt/chatbot/shared/classifier_embeddings.json \
       /opt/chatbot/current/ai-core/data/eval/classifier_embeddings.json
```

**What happens if it's missing:** the classifier is now **warmed at startup**
(not on the first user message), so a missing/stale cache **no longer freezes
a live request** — instead the backend just takes ~30s longer to finish
booting (it re-embeds the exemplars at boot). The symlink keeps that boot warm
fast (seconds instead of ~30s) and avoids re-paying it every restart. Preflight
(step 3) flags a missing cache.

### 2. Run the deploy script
Rebuilds the backend, **rebuilds the React frontend** (`npm run build`), swaps
the `current` symlink, restarts the service.
```bash
/opt/chatbot/deploy.sh
```

### 3. Preflight check (gate before/after start)
Validates the embedding cache, env keys, nginx, Weaviate, and Postgres.
```bash
ROOT=/opt/chatbot/current bash /opt/chatbot/current/ai-core/scripts/preflight.sh
```

### 4. Post-deploy smoke test (two-turn Socket.IO conversation)
```bash
bash /opt/chatbot/current/ai-core/scripts/post_deploy_check.sh
```

### 5. Confirm it's up
```bash
curl -s http://localhost:8081/health/live      # -> {"status":"alive"}
curl -s http://localhost:8081/health/ready | jq # per-dependency status
```

## Post-deploy verification

| Check | Expected |
|---|---|
| Startup log | Shows `🛫 [Springshare] pre-flight health check` with LibCal/LibGuides ✅ |
| Startup log | Shows `✅ [v2] serving deps warmed` before `🚀 Application startup complete` (boot warm ran) |
| First message after restart | Answers normally — **no "I encountered an error" / dropped connection** (cold-start freeze fixed) |
| Startup log | **No** `ManualCorrection load failed ... bound to a different event loop` (D3 fixed) |
| Runtime log | After asking a question, you see `[Springshare:LibCal]` / `🤖 [OpenAI]` lines |
| `/health/ready` | `postgres`, `weaviate`, `openai`, `libcal`, **`libguides`** all present, status healthy |
| Browser (hard refresh, **new conversation**) | A small **"Sources"** list with **clickable** blue links appears under each answer |
| Ask "Does the King MakerSpace have a 3D printer?" | "Yes …" citing the MakerSpace page (not a refusal) |
| Ask "address of the Art library" | `Alumni Hall, Oxford, OH 45056` (data already live) |

> The Sources footer only appears after a **hard refresh (Ctrl/Cmd+Shift+R)**
> and in a **new conversation** — old answers are static text and won't change.

## How to view logs

Logs go to **journald** under the `smartchatbot-backend` unit.

```bash
# Live tail (most common)
sudo journalctl -u smartchatbot-backend -f

# Last 200 lines
sudo journalctl -u smartchatbot-backend -n 200 --no-pager

# Only since the last restart
sudo journalctl -u smartchatbot-backend --since "5 min ago" --no-pager

# Warnings and errors only
sudo journalctl -u smartchatbot-backend -p warning -n 100 --no-pager
```

### Targeted filters for this release
```bash
# Pre-flight Springshare banner (printed once at startup)
sudo journalctl -u smartchatbot-backend --since "10 min ago" | grep -A4 "Springshare.*pre-flight"

# Watch all external API traffic live (Springshare + OpenAI)
sudo journalctl -u smartchatbot-backend -f | grep -E "Springshare|OpenAI"

# Confirm the D3 fix: this line should NO LONGER appear
sudo journalctl -u smartchatbot-backend --since "10 min ago" | grep "ManualCorrection load failed"
#   (no output = fixed; if it still appears, the deploy/cache is wrong)

# Confirm the classifier loaded fast (didn't hang)
sudo journalctl -u smartchatbot-backend --since "10 min ago" | grep -i "exemplar"
```

### If the unit name differs on this host
```bash
systemctl list-units --type=service | grep -iE "chatbot|smartchat"
```
Use the real name in place of `smartchatbot-backend`.

### If logs are written to a file instead of journald
```bash
sudo find /opt/chatbot /var/log -name "*.log" -newermt "-1 day" 2>/dev/null | grep -i chat
# then: sudo tail -f <that file>
```

### Recommended: watch the startup sequence live
```bash
sudo journalctl -u smartchatbot-backend -f
```
You should see, in order:
`🚀 Application starting` → Database ✅ → Weaviate ✅ →
`🛫 [Springshare] pre-flight health check` → `🚀 Application startup complete`.

## Rollback

`deploy.sh` keeps the last 5 builds. To roll back, repoint the `current`
symlink to the previous build dir and restart:
```bash
ls -dt /opt/chatbot/builds/*/ | head -5          # list recent builds
ln -sfn /opt/chatbot/builds/<PREVIOUS_TS> /opt/chatbot/current
sudo systemctl restart smartchatbot-backend
```
Data changes (Postgres) are not affected by a code rollback.
