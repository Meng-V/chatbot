# Smart Chatbot — ideal `deploy.sh` + nginx (spec for Puppet / Server Admin)

**Audience:** whoever owns the Puppet templates for `ulblwebp20` (`/opt/chatbot`).
**Author:** Meng. **Goal:** automate the whole "To start" runbook into one
`deploy.sh`, **gate before going live**, and **roll back automatically** if the
new build can't serve. Puppet only needs to template the script + the nginx
`location` block below — no logic lives in Puppet.

---

## 1. What changes vs. today, and why

| | Today's `deploy.sh` | Ideal (this spec) |
|---|---|---|
| Order | clone → build → **flip `current` + restart** → done | clone → build → **gate** → flip + restart → **verify** → prune |
| Checks | none | `preflight.sh` + `pre_deploy_smoke.py` **before** flip; `post_deploy_check.sh` **after** |
| Bad build | **goes live, users see errors** | **never flips** — old build keeps serving, deploy exits non-zero |
| If it breaks after going live | stays broken until someone notices | **auto-rollback** to the previous build + restart |
| Embeddings symlink | created on `current/` after flip | created on the new build **before** the gates (the classifier needs it) |

The three gate scripts already exist in the repo and **all exit non-zero on
failure**, so the script just checks exit codes — no string-grepping:

- `ai-core/scripts/preflight.sh` — env / weaviate / postgres / embedding cache / dist / dead-URL guard. Honors `ROOT`, `PORT`, `ENV_FILE`.
- `ai-core/scripts/pre_deploy_smoke.py` — replays the **real multi-turn serving path** (the thing that kept breaking on 2nd message). Must run as `smartchatbot` (it connects Prisma; the engine binary is owned by `smartchatbot`).
- `ai-core/scripts/post_deploy_check.sh` — hits the **live** service (`/health/live` + a real socket.io two-turn).

## 2. The ideal `deploy.sh`

```bash
#!/usr/bin/env bash
# Template for deploy.sh — generated by Puppet.
# Smart Chatbot deploy: build → gate → go-live → verify → rollback-on-failure.
set -euo pipefail

APP_DIR=/opt/chatbot
REPO=https://github.com/Meng-V/chatbot.git
PORT=8081
SERVICE=smartchatbot_backend.service        # <-- Puppet: confirm exact unit name
TS=$(date +%Y%m%d%H%M%S)
RELEASE_DIR="$APP_DIR/builds/$TS"

# Branch arg (defaults to main; Puppet may override the default via hiera).
if [ "$#" -lt 1 ]; then BRANCH=main; else BRANCH="$1"; fi
echo "==> Deploying branch '$BRANCH' into $RELEASE_DIR"

# Remember where 'current' points NOW, for rollback if the new build fails.
PREV_TARGET=""
[ -L "$APP_DIR/current" ] && PREV_TARGET="$(readlink -f "$APP_DIR/current")"

run_as_app() { sudo -u smartchatbot bash -lc "$1"; }

rollback() {
  echo "↩️  ROLLING BACK to ${PREV_TARGET:-<none>}"
  if [ -n "$PREV_TARGET" ]; then
    sudo -u smartchatbot ln -sfn "$PREV_TARGET" "$APP_DIR/current"
    sudo /bin/systemctl restart "$SERVICE"
    echo "   rolled back; previous build is serving again."
  else
    echo "   no previous build recorded — manual intervention needed."
  fi
}

# ---------------------------------------------------------------------------
# PHASE 1 — BUILD the new release (NOT live yet; users stay on the old build)
# ---------------------------------------------------------------------------
sudo -u smartchatbot mkdir -p "$RELEASE_DIR"
sudo -u smartchatbot git clone --branch "$BRANCH" --depth 1 "$REPO" "$RELEASE_DIR"

# env + embedding-cache symlinks. The embedding cache MUST exist before the
# gates run — the kNN classifier loads it (a clean build without it hangs the
# first message 30-60s). Both owned by smartchatbot.
sudo -u smartchatbot ln -sfn "$APP_DIR/shared/.env" "$RELEASE_DIR/.env"
sudo -u smartchatbot mkdir -p "$RELEASE_DIR/ai-core/data/eval"
sudo -u smartchatbot ln -sfn "$APP_DIR/shared/classifier_embeddings.json" \
    "$RELEASE_DIR/ai-core/data/eval/classifier_embeddings.json"

# backend venv + frontend dist
run_as_app "
  export NVM_DIR=\"\$HOME/.nvm\"
  [ -s \"\$NVM_DIR/nvm.sh\" ] && . \"\$NVM_DIR/nvm.sh\"
  cd '$RELEASE_DIR/ai-core'
  python3.13 -m venv venv
  source venv/bin/activate
  pip install -e .
  prisma generate
  cd '$RELEASE_DIR/client'
  npm ci
  npm run build
"
echo "==> Build complete"

# ---------------------------------------------------------------------------
# PHASE 2 — PRE-FLIP GATES (run against the NEW build; nothing live changed).
#   Any failure ABORTS the deploy. Old build keeps serving. Fix and re-run.
# ---------------------------------------------------------------------------
echo "==> Gate 1/2: preflight (env, weaviate, postgres, embeddings, dist, URLs)"
if ! run_as_app "cd '$RELEASE_DIR' && ROOT='$RELEASE_DIR' PORT=$PORT bash ai-core/scripts/preflight.sh"; then
  echo "❌ PREFLIGHT FAILED — deploy aborted; 'current' still on previous build."
  echo "   Read the ✗ lines above, fix, then re-run deploy.sh."
  exit 1
fi

echo "==> Gate 2/2: pre-deploy smoke (real multi-turn serving path)"
if ! run_as_app "cd '$RELEASE_DIR' && ai-core/venv/bin/python ai-core/scripts/pre_deploy_smoke.py"; then
  echo "❌ PRE-DEPLOY SMOKE FAILED — deploy aborted; 'current' still on previous build."
  exit 1
fi

# ---------------------------------------------------------------------------
# PHASE 3 — GO LIVE: flip 'current', restart, wait for health.
# ---------------------------------------------------------------------------
echo "==> Gates passed. Switching 'current' -> new build and restarting."
sudo -u smartchatbot ln -sfn "$RELEASE_DIR" "$APP_DIR/current"
sudo /bin/systemctl restart "$SERVICE"

echo "==> Waiting for backend /health/live ..."
UP=0
for _ in $(seq 1 30); do
  if curl -fsS "http://localhost:$PORT/health/live" >/dev/null 2>&1; then UP=1; break; fi
  sleep 2
done
if [ "$UP" -ne 1 ]; then
  echo "❌ Backend did not become healthy within 60s."
  rollback
  exit 1
fi

# ---------------------------------------------------------------------------
# PHASE 4 — POST-DEPLOY VERIFY against the LIVE service. Fail => roll back.
# ---------------------------------------------------------------------------
echo "==> Smoke: /smoketest/v2 (informational)"
curl -fsS "http://localhost:$PORT/smoketest/v2" || echo "  (smoketest/v2 non-200 — see post-deploy check below)"

echo "==> Post-deploy check (live socket.io two-turn)"
if ! sudo /bin/bash "$APP_DIR/current/ai-core/scripts/post_deploy_check.sh"; then
  echo "❌ POST-DEPLOY CHECK FAILED on the live build."
  rollback
  exit 1
fi

# ---------------------------------------------------------------------------
# PHASE 5 — keep only the 5 most recent builds.
# ---------------------------------------------------------------------------
cd "$APP_DIR/builds" && sudo ls -1dt */ | tail -n +6 | xargs -r rm -rf

echo "✅ DEPLOY SUCCESSFUL — now serving build $TS ($BRANCH)"
```

### Why this order is the important part
- **Gates run on the new build BEFORE `current` flips.** A broken build can
  never reach users — the worst case is "deploy aborted, nothing changed."
- **`pre_deploy_smoke.py` runs the same path the socket handler runs** (classify
  → agent in an executor thread → synth → conversation_store), which is exactly
  where the "works locally, 500s on the 2nd message on prod" bugs lived. If that
  class of bug comes back, the deploy stops here instead of in front of a librarian.
- **Post-deploy failure auto-rolls-back.** If the live build somehow fails the
  socket two-turn after restart, it flips back to the previous build instead of
  leaving users with a broken bot.

## 3. nginx — the `/ticket/create` block (so ticket POST works)

**The problem:** the ticket form POSTs to `/ticket/create`. `/smartchatbot/` is a
**static alias** (`alias /opt/chatbot/current/client/dist/`) — it only serves
files, so any POST under it returns **405**. The backend's `/ticket/create` route
lives at the **root** path and must be reverse-proxied, exactly like the already
-working `/summarize-chat`.

**The fix:** in the `app.lib.miamioh.edu` server block, add a `location
/ticket/create` that mirrors the existing `/summarize-chat` block. Easiest and
safest: **copy the `/summarize-chat` block verbatim and change only the path.**
Reference shape:

```nginx
# Backend API endpoints reverse-proxied to the FastAPI upstream.
# These are at the ROOT path, NOT under the /smartchatbot/ static alias.
location = /ticket/create {
    proxy_pass         http://smartchatbot_backend;   # same upstream as /summarize-chat
    proxy_http_version 1.1;
    proxy_set_header   Host              $host;
    proxy_set_header   X-Real-IP         $remote_addr;
    proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header   X-Forwarded-Proto $scheme;
    proxy_read_timeout 30s;
}
```

Notes for Server Admin:
- Match the proxy headers of the **sibling `/summarize-chat` block** rather than
  trusting the list above — keep them identical so all backend endpoints behave
  the same.
- The full set of root endpoints the backend serves (all proxied the same way):
  `/summarize-chat`, **`/ticket/create`**, `/askus-hours/status`, `/health`,
  `/health/live`, `/health/ready`, `/smoketest`, `/smoketest/v2`.
- `/smartchatbot/socket.io/` stays as its own proxied (websocket-upgrade) block.
- After editing, `sudo nginx -t && sudo systemctl reload nginx`.

### Optional cleanup (future, not for beta)
Adding one `location` per endpoint is why the config feels messy. If you want to
stop editing nginx every time the backend gains an endpoint, route them as a set:

```nginx
location ~ ^/(summarize-chat|ticket/create|askus-hours|health|smoketest) {
    proxy_pass http://smartchatbot_backend;
    # ...same headers...
}
```

This is a behavior change to matching precedence, so do it deliberately (test
with `nginx -t` + a real POST), not as part of the ticket fix.

## 4. Puppet / ops notes

- **Confirm the systemd unit name.** The current `deploy.sh` restarts
  `smartchatbot_backend.service` (underscore); some docs say
  `smartchatbot-backend` (hyphen). Set `SERVICE=` once at the top of the script
  to whichever is real.
- **Gates run as `smartchatbot`**, not root — `pre_deploy_smoke.py` connects
  Prisma and the query-engine binary is owned by `smartchatbot` (running it as
  another user → PermissionError). `post_deploy_check.sh` runs under `sudo`
  because it talks to the live service only.
- **Don't drop the embedding-cache symlink.** It's a ~340 MB file kept in
  `/opt/chatbot/shared/`, symlinked into each build. A clean build without it
  doesn't crash — it silently hangs the first message — so `preflight.sh`
  treats a missing/small cache as a hard failure.
- **Exit codes are the contract.** Non-zero from `deploy.sh` = do not consider
  the deploy done. Every gate already returns non-zero on failure.
- This script is **idempotent and safe to re-run**: a failed run before the flip
  leaves `current` untouched; a failed run after the flip rolls back.

## 5. The old manual runbook → now automatic

Everything in the Google-doc "To start" (deploy → pre_deploy_smoke → preflight +
embeddings symlink → post_deploy_check + smoketest → log view) is folded into the
script above. The only manual steps left are the **exceptions**:

- **Reverting further back** (more than one build): `sudo ln -sfn
  /opt/chatbot/builds/<TS> /opt/chatbot/current && sudo systemctl restart $SERVICE`.
- **`.env` changes**: edit `/opt/chatbot/shared/.env`, then re-deploy or restart.
- **Weaviate/docker**: `sudo systemctl restart docker-weaviate` (unchanged).
